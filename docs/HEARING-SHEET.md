# CareNote AI 導入ヒアリングシート

> コンサル先（特養・小規模多機能等の運営企業）との打ち合わせで使う質問リスト。
> 目的は4つ: **(A) AIに渡す「ベテランの知見」の収集**（品質エンジンの素材・最重要）/
> **(B) 自動入力(P2)の実現可否判断**（介護ソフトの特定）/ **(C) 効果測定の物差し作り**（時間の実測）/
> **(D) 個人情報の取り扱い合意**。
> 回答は本リポジトリでは管理せず、コンサル先との共有フォルダ等で管理する（個人情報・社外秘を含むため）。

---

## 0. 進め方（吉本さん向けメモ）

- 1回で全部聞こうとしない。優先順位は **1 → 2 → 5 → 3 → 4 → 6**（最初の訪問は 1・2・5 だけでも成立）。
- 「正しい答え」を聞くのではなく「現場の実態」を聞く。建前の手順より、実際にやっていることが素材になる。
- §5（ベテランのコツ）は録音許可をもらうと後の言語化が楽（許可は必ず書面 or 記録に残す）。

## 1. 基本情報・体制

- [ ] 対象事業所（居宅介護支援/特養/小多機 …）と拠点数
- [ ] ケアマネジャーの人数・経験年数の分布（ベテラン◯名 / 中堅◯名 / 新人◯名）
- [ ] 1人あたりの担当利用者数（平均・最大）
- [ ] 書類作成は誰がいつやっている?（訪問の合間 / 夕方まとめて / 残業 / 持ち帰り）
- [ ] 今いちばん困っている書類業務は何か（自由回答・最初に聞く）

## 2. 使っている介護ソフト（→ P2 自動入力の要）

- [ ] ソフト名・提供元（例: カイポケ / ほのぼの / ワイズマン / 絆 / その他: ）
- [ ] 利用形態: ☐ クラウド（ブラウザで使う） ☐ インストール型（PCアプリ） ☐ 併用
- [ ] ブラウザで使う場合: 対応ブラウザ（Chrome / Edge / それ以外）
- [ ] 帳票（アセスメント・第1〜3表・モニタリング）はソフト内で作成? それとも Excel/Word/手書き?
- [ ] CSV・データの出力（エクスポート）機能はあるか / 使ったことがあるか
- [ ] 複数ソフトを使い分けている場合、その内訳
- [ ] （後で当社側で確認）ソフトの利用規約上、拡張機能・自動入力の可否 ※要規約確認

## 3. 様式・帳票の現物（→ 生成フォーマットを実物に合わせる）

- [ ] **現物のコピーを入手**（個人情報をマスキングした記入済み例が最良。空様式でも可）
  - [ ] アセスメント様式（課題分析標準項目23項目? 独自様式? パッケージ準拠?）
  - [ ] 居宅サービス計画書 第1表・第2表（・第3表）
  - [ ] モニタリング様式
  - [ ] 支援経過記録（第5表）※今回スコープ外だが参考に
- [ ] 実地指導・監査で過去に指摘された記載事項はあるか（→ ルール化の最優先素材）
- [ ] 法人独自の記載ルール・マニュアルはあるか（あれば入手）

## 4. 現状の作成時間（→ KPI のベースライン実測）

「だいたいの感覚」で良いので1件あたりの時間を聞く。可能なら後日2〜3名に1週間だけ実測を依頼。

|---|---|---|---|
| アセスメント | 分 | 件 | |
| ケアプラン第1・2表（新規） | 分 | 件 | |
| ケアプラン第1・2表（更新） | 分 | 件 | |
| モニタリング記録 | 分 | 件 | |

- [ ] 書類業務に起因する残業・持ち帰りの実態
- [ ] 「時間がかかる工程」はどこか（情報整理? 文章化? 転記? 印刷・配布?）

## 5. ベテランの「コツ」の抽出（→ 品質エンジンの素材）

> **注**: 基本の型は吉本（ケアマネ歴13年）の知見で `lib/rules/` v1 に反映済み
> （意向形ニーズ／長期12か月・短期6か月／です・ます体／アセスメント連動・根拠の記録性）。
> 本節はコンサル先**固有**の流儀・差分の確認用。先方のベテラン＋できれば新人1名に、別々に聞く（差分が「暗黙知」）。

### 良し悪しの基準
- [ ] 「良いケアプラン」と「惜しいケアプラン」の実例を1つずつ見せてもらう（マスキング済みで）
- [ ] その2つの**どこが違うのか**を言葉にしてもらう
- [ ] 新人のプランを直すとき、よく直すポイント TOP3
- [ ] 第2表のニーズの書き方で「これはダメ」という型（例:「〜できない」の羅列 等）

### 書き方の型
- [ ] ニーズ・目標でよく使う言い回し / 避ける言い回し（言い換え集があれば最高）
- [ ] 長期・短期目標の期間の決め方の目安
- [ ] インフォーマル支援（家族・地域）をどう位置づけているか
- [ ] 意向欄: 本人の言葉をどこまでそのまま書くか
- [ ] アセスメントで必ず確認する項目・見落としやすい項目
- [ ] モニタリングで「漫然と継続」にしないための判断基準

### 確認・チェック
- [ ] 提出前にセルフチェックしている観点（チェックリストがあれば入手）
- [ ] 管理者・主任ケアマネがレビューで見ている観点

## 6. 個人情報・セキュリティ（→ 導入可否・SPEC §7）

- [ ] 個人情報取扱規程の有無（あれば入手）。外部クラウド/AIサービス利用に関する定めはあるか
- [ ] 利用者・家族からの個人情報利用同意の範囲（外部委託・システム利用がどう書かれているか）
- [ ] AI（海外サーバ含む）へ送るデータについての会社としての方針・懸念
- [ ] 氏名を記号化（仮名化）した運用なら許容できるか
- [ ] データ処理に関する覚書・契約の要否（誰が判断者か）

## 7. 実証（トライアル）の進め方

- [ ] 試してくれるケアマネ候補（前向きな人1〜2名から）
- [ ] 使用PC・ブラウザ環境（Chrome が使えるか）
- [ ] フィードバックの頻度・方法（週1の15分オンライン 等）
- [ ] 成功の判断基準のすり合わせ（例: ◯か月で作成時間◯%減＋品質チェック合格率）
- [ ] 経営側の期待値（時間削減? 品質平準化? 採用・定着? 監査対応?）→ 優先順位を1つに

## 持ち帰りチェックリスト（ヒアリング後にやること）

1. §5 の回答を `lib/rules/`（assessment / carePlan / monitoring）のルールに翻訳して差し替え
2. §3 の現物様式と生成フォーマット（types/）の項目差分を洗い出し
3. §2 のソフト名で P2（ブラウザ拡張）の技術調査・規約確認
4. §4 の数値で SPEC §6.4 の KPI を確定
5. §6 の回答を SPEC §7 に反映（必要なら覚書ドラフト作成）
