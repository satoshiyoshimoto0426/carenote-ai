## ローンチを止める関門（ゲート）は4つ

1. **G1 法務・個人情報の契約整備**（P-LEGAL-A）── 要配慮個人情報を海外API（Claude）へ送る構成の法的整理
2. **G2 カイポケ実機検証＋規約照会**（P-KAIPOKE）── 実画面で流し込みが動くこと＋SMS社照会
3. **G3 拡張API受け口の認可是正**（P-SECURITY）── CORS `*`×単一トークンの是正
4. **G4 出口の総合判定**（P-LAUNCH）── G1〜G3緑＋KPIベースライン＋人間承認のデータモデル強制

### 【並行レーン2・実証前に必須】P-KPI｜合否基準を先に決める
- **実証開始前に**現行の作成時間・品質バラつきをベースライン計測 👤🏢
- 成功閾値を先方と数値合意（例: 1件30分→15分／評価スコア分散30%減）🏢👤
- ベースラインは**2条件**（①コピペ運用 ②拡張流し込み）── 拡張が規約NGでもコピペ単体の価値を数値で示せる保険

### 【並行レーン3・背骨】K1｜知見エンジン Lite（Obsidian → lib/rules）
- 吉本さんの新方針「第1〜7表の作成ポイント＋公式ルールブックをナレッジ化→AIが準拠して作成」を §2.5-F 準拠で段階化:
  - **Lite（今すぐ）**: Obsidianで執筆 → 帳票別mdを `lib/rules/` に手動反映。技術投資ゼロ
  - **Structured（実証中）**: 「帳票→ルールファイル」の静的マッピングで該当分のみ注入（プロンプトキャッシュ活用）。動的検索・ベクトルDBは**作らない**
  - **昇格（対象外・grepで追えない規模になったら）**: embedding検索を検討
- 言語化テンプレ: 1ルール＝「状況／禁止／推奨／根拠」👤🧩。反映の担当・頻度を運用として固定（**律速は技術でなく吉本さんの言語化時間**）
- 公式文書の扱い（著作権）: **法令・告示・通達は著作権法13条で取込可**。**白書・手引き・マニュアルは著作権あり→全文転載せず要旨をルール化＋出典明示** 🧩
- 品質ゲート: ルール更新のたび評価エンジンで回帰採点（P-EVAL）

### P-GEN｜生成品質の実データ確定（実証帳票を1〜2種に絞る）

### P-KAIPOKE｜カイポケ実機検証＋規約照会 ★G2

### P-LAUNCH｜1社実証ローンチ ★G4
- **人間承認をデータモデルで強制**: documents の status に `approved` を追加し、人間操作なしに承認不可・未承認はコピー/流し込み不可 🧩
- 価値訴求は「作成時間短縮＋記載漏れ検知の**補助**」に限定（「監査リスク低減の保証」とは謳わない）👤

- KPI の after 計測→ベースラインと比較→先方と合否判定 👤🏢

## 最重要リスク

1. **R-LEGAL**: DPA/ZDR の所要日数が未取得でローンチ日が逆算できない（→P-LEGAL-Aの照会が全体の第一手）

3. **R-KAIPOKE**: 規約NG/実機NGなら時短エンジンがコピペ縮退（→2条件KPIで価値を担保）
4. **R-KNOWLEDGE**: 知見エンジンの律速は吉本さんの言語化時間（→テンプレと反映運用の固定で軽減）
5. **R-RESPONSIBILITY**: 「AIが制度準拠を判断」は保証できない → 承認データモデル＋訴求限定＋人間の最終確認で設計として担保
