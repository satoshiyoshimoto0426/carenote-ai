## 0. この再定義のひとことサマリ

> **ケアマネジャーの「書類作成」を、AIが“ベテランの型”で下書きし、ブラウザ拡張で既存の介護ソフトに自動入力する。** 「時間がかかる」と「品質が人によってバラつく」を同時に解く。

旧プロダクト（PDFをアップロードして採点するだけ）から、本来の目的だった **記録・帳票の作成補助** へ軸を戻す。採点機能は独立機能として残す。

## 1. 背景・目的（Why）

### 1.1 きっかけ
特別養護老人ホーム・小規模多機能型居宅介護などを経営する企業から、**AI業務改善コンサル**を依頼された。その中核プロダクトとして CareNote AI を据える。

### 1.2 解決する2大課題と、対応する2つのエンジン
本プロダクトの設計思想は「**2つの課題 → 2つの仕組み**」という1対1対応にある。

| 解決したい課題 | それを担う仕組み（エンジン） | ねらい |
|---|---|---|
| ① 書類作成に **時間がかかりすぎる** | **時短エンジン**：ブラウザ拡張で既存の介護ソフトに自動入力（＋面談メモ・音声・既存データを材料化） | 転記・入力の手作業をなくす |
| ② 品質が **個人の能力に依存** し一定にならない | **品質エンジン**：「作成のコツ・方法・守るべきルール」を言語化・構造化してAIに渡す（ベテランの暗黙知のルール化） | 誰がやっても一定品質の下書きを生む |

この **②品質エンジン（ナレッジのルール化）こそが製品の知的な核** であり、競合に対する差別化点になる。①の自動化は「速さ」を、②のルール化は「質」を担保する。

### 1.3 やらないこと（このプロダクトの非目標）
- 介護ソフトそのものを置き換える（あくまで既存ソフトに後付けで価値を足す）。
- ケアマネジャーの判断を AI が代替する（AIは下書きまで。**確定・提出は必ず人間**）。

## 2. ターゲットユーザーと提供価値（Who / Value）

- **最初のユーザー**：ケアマネジャー（介護支援専門員）。居宅介護支援事業所で、利用者のケアプラン関連書類を作成する人。
- **拡張先（将来）**：特養・小多機などの介護記録・帳票、生活相談員・介護職員など他職種。
- **提供価値**：
  1. 書類1件あたりの作成時間を大幅短縮（※KPIは第6章）。
  2. 事業所内で**品質を平準化**（新人でもベテランの型で書ける）。
  3. 記載漏れ・整合性チェックを内蔵し、**監査リスクを低減**。

## 3. プロダクト全体像（What）

### 3.1 対象書類（MVPスコープ）
ケアマネジメントの一連の流れに沿って、次の3工程を対象にする（支援経過記録〔第5表〕は今回見送り）。
**2026-06-11 追記**: 吉本の判断により **第4表（サービス担当者会議の要点）** を対象に追加（会議メモ→要点の構造化）。第6・7表（給付管理・単位数計算）は計算業務のため対象外とする。
**2026-06-11 追記2**: 当初見送りとした **第5表（居宅介護支援経過）** も対象に追加（対応メモ→「対応内容/背景/事実・発言/判断/今後」の5層記録。実地指導対策の要のため）。

```
① アセスメント・課題分析   →   ② ケアプラン本体（第1・2表）   →   ③ モニタリング記録
   利用者の状態を分析し           総合方針・生活課題(ニーズ)・            毎月の見直し。目標達成度を
   課題を抽出する前段             目標・サービス内容を記載              確認し継続要否を判断
```

> ※要確認：各書類の正式様式・項目（例：課題分析標準項目、第1表/第2表のレイアウト、モニタリングの記載要件）は、コンサル先が実際に使っている様式に合わせて確定する。本SPECでは「書類の種類」レベルで合意し、項目詳細は実装前に現物で詰める。

### 3.2 4つのコンポーネント

```
┌─────────────────────────────────────────────────────────────┐
│  A. ナレッジ・ルールベース（品質エンジン）                    │
│     「作成のコツ・方法・守るべきルール」を構造化して保持        │
│        ↓ AIに渡す                                            │
│  B. 下書き生成バックエンド（Next.js / Claude API）            │
│     入力（メモ/音声/既存データ）＋ルール → 帳票の下書きを生成   │
│        ↓ 生成結果                                            │
│  C. ブラウザ拡張機能（時短エンジン）                          │
│     既存の介護ソフトの画面を読み、欄に自動入力（人が最終確認）  │
│                                                              │
│  D. 評価・採点（独立機能・既存資産を継承）                     │
│     既存の書類PDFを8カテゴリで点検する単独機能                 │
└─────────────────────────────────────────────────────────────┘
```

- **A. ナレッジ・ルールベース**：本プロダクトの核。Markdown＋構造化データでルールを管理し、AIのプロンプトに注入する。ベテランの知見を増やすほど品質が上がる「はずみ車」。
- **B. 下書き生成バックエンド**：既存の Next.js アプリを継承・発展。入力を受け取り、ルールに従って各帳票の下書き（構造化テキスト/JSON）を生成。
- **C. ブラウザ拡張機能**：新規コンポーネント。Chrome/Edge 拡張。介護ソフトの画面（DOM）を読み、生成内容を該当欄へ転記・自動入力。**最初は半自動（人が確認して確定）**。
- **D. 評価・採点**：現行アプリの機能をそのまま独立機能として維持。開発時には「生成した下書きの品質チェック（回帰テスト）」にも転用できる（社内利用）。

### 3.3 入力（AIへの材料）
1. **面談メモ・アセスメント結果の入力**（MVPの主役。手入力／貼り付け）
2. **音声 → 文字起こし → 整形**（Phase 3）
3. **既存ソフトの出力（CSV/PDF等）の取り込み**（Phase 3）
4. **画面からの読み取り**（拡張機能が既存ソフトの既存データを材料に）

## 4. 段階的フェーズ（How / いつ何を作るか）

リスクの小さい「品質エンジン」を先に出し、難所の「ブラウザ自動化」は対象を絞って後段に置く。**まず1社で実証 → 後で汎用化**の方針に沿う。

| フェーズ | 内容 | 価値が出る単位 | 主なリスク |
|---|---|---|---|
| **P0 資産整理** | 未コミット変更の整理、評価機能を独立機能として整理、docs/CONTEXT-MAP 作成、テスト基盤(vitest)導入 | 土台が安定 | 小 |
| **P1 品質エンジン**（最重要） | ルールベース(A)＋下書き生成(B)。入力は手入力メモ。アセス/第1・2表/モニタリングの**下書きテキストを生成**。Webの管理画面で確認・編集・コピー | **自動入力なしでも「一定品質の下書き」で価値実証** | 小〜中 |
| **P2 時短エンジン** | ブラウザ拡張(C)で、生成内容を**対象ソフト1つ**の画面に半自動入力（人が確認して確定）。**対象＝カイポケ（一旦・2026-06-11吉本決定。調査: docs/P2-KAIPOKE.md）** | 「転記の手間ゼロ」を実証 | **大**（対象ソフト依存・規約・画面変更） |
| **P3 入力拡充** | 音声→文字起こし、既存ソフト出力の取り込み | 入力の手間も削減 | 中 |
| **P4 汎用化** | 複数ソフト対応、複数事業所向けSaaS化、個人情報・セキュリティ強化、課金 | 横展開 | 中〜大 |

> 設計判断：**P1 は P2 が無くても単体で価値が出る**ように作る（下書きを画面に出してコピペでも時短になる）。これにより「ブラウザ自動化が難航しても製品が死なない」状態を担保する。

## 5. アーキテクチャ・技術スタック

### 5.1 方針
- 既存の Next.js リポジトリを**継承・発展**させる（作り直さない）。
- 既存スタックを維持：**Next.js 16 / React 19 / TypeScript / Biome / Tailwind CSS 4 / Supabase / Clerk / Claude API**。
- 新規追加：**ブラウザ拡張**（Chrome/Edge, Manifest V3, TypeScript）。
- **特定の介護ソフトに依存しない**設計（対象ソフトが未確定・複数のため）。拡張は「サイトごとのアダプタ（どの欄に何を入れるか）」を差し替え可能な構造にする。

### 5.2 コンポーネント別スタック

| コンポーネント | 技術 | 補足 |
|---|---|---|
| A. ルールベース | Markdown ＋ 構造化データ（JSON/TS） | `lib/rules/` に配置。AIプロンプトへ注入 |
| B. 下書き生成 | Next.js API Routes ／ Server Actions ＋ Claude API | 用途別にモデル選定（生成＝高品質モデル、補助＝軽量モデル）。最新の Claude モデルを使用 |
| C. ブラウザ拡張 | Manifest V3 ＋ TypeScript（content script / background） | `extension/`（新規）。バックエンドの生成APIを呼ぶ |
| D. 評価・採点 | 既存実装（`app/api/evaluate` 等）を継承 | モデル/ヘッダーは最新仕様へ更新（※実装時に公式ドキュメントで検証） |
| 保存 | Supabase（RLS必須） | 個人情報の保存は最小限・暗号化方針（第7章） |
| 認証 | Clerk | 既存 |

## 6. 主要機能と受け入れ条件（Acceptance Criteria）

### 6.1 P1：下書き生成（最優先）
- **F1 ルールベース管理**：作成ルール（コツ・禁止事項・記載例）を構造化して保持できる。
  - 受け入れ：ルールを追加/編集すると、生成結果に反映される。ルールは人が読める形（Markdown）で一覧できる。
- **F2 アセスメント下書き生成**：面談メモを入力すると、課題分析の下書きを生成する。
  - 受け入れ：入力に対し、空欄を勝手に創作せず（不明は「要確認」と明示）、ルールに沿った下書きが出る。
- **F3 ケアプラン第1・2表 下書き生成**：アセス結果から、総合方針・生活課題（ニーズ）・長期/短期目標・サービス内容の下書きを生成する。
  - 受け入れ：第2表のニーズ→目標→サービスの**整合**が取れている。配点・必須項目の記載漏れがない。
- **F4 モニタリング下書き生成**：前回プランと最新状況から、モニタリング記録の下書きを生成する。
- **F5 確認・編集・出力**：生成結果を画面で確認・編集し、コピー／（将来）拡張へ受け渡しできる。
  - 受け入れ：**人間が編集・承認してからでないと「確定」できない**（AI出力は常に下書き扱い）。

### 6.2 P2：ブラウザ拡張で自動入力
- **F6 画面認識**：対象介護ソフトの帳票画面で、入力欄を認識する。
- **F7 半自動入力**：生成内容を該当欄に入力する。**入力後、人が確認して保存/確定**する（全自動確定はしない）。
  - 受け入れ：誤った欄への入力がない。確定前に差分（何を入れたか）を人が確認できる。

### 6.3 D：評価・採点（独立機能・継承）
- **F8 書類評価**：既存書類PDFを8カテゴリで採点し、改善点を出す（現行どおり）。

### 6.4 成功指標（KPI・※コンサル先と合意）
- 書類1件あたりの作成時間：**◯分 → ◯分**（※実測ベースラインを取って設定）。
- 事業所内の品質ばらつき（評価スコアの分散）の縮小。
- ケアマネの主観的負担（アンケート）の改善。

### 6.5 救済モード（書類一式の一括下書き）
> 追加: 2026-06-16（吉本提案・決定）。実装: `app/(dashboard)/rescue/` ＋ `app/api/rescue/` ＋ `lib/generation/rescue.ts`。

- **F9 人物像ドリブンの一式生成**：利用者の人物像を**構造化フォーム**（基本情報／性格・人柄／生活歴／
  既往・診断・服薬／心身の状態／家族・住環境／利用サービス／本人・家族の意向／補足）に手打ち入力すると、
  人物像・サービス・目標が**一貫した書類一式**（アセスメント → 第1・2表 → 第4表 → 第5表 → モニタリング）を
  AIがまとめて下書きする。第6表（給付管理票）は対象外（従来どおり）。
  - 入力は全項目任意で、1項目以上あれば生成可（空欄はAIが想定して補う）。基本情報以外を
    `composePersonaNotes` でラベル付きメモに合成して生成器へ渡す。
  - 仕組み：依存順に生成し上流の出力を下流の入力に渡す（アセス結果→第1表、ケアプラン→会議/経過/モニタリング）。
  - 受け入れ：全帳票で人物像・利用サービス・目標が矛盾しない。
- **品質ルールの限定的緩和（吉本さん＝ケアマネ歴13年 承認済み・横断規約 §2.7-C）**：
  救済モードに限り「不明は創作せず要確認」を緩め、不足項目もAIが想定して**完成形まで埋める（項目単位の保留印なし）**。
  - 緩和は救済モードのみ。通常の `/create`・`/api/generate` は従来ルール（創作しない）を維持する。
  - **大原則は維持**：AI出力は下書きであり、確定前に人間が**事実と照合**して確認・修正する（§12 Always）。
    項目単位の印は省くが、モード冒頭のバナーでこの前提を明示する。
  - 氏名・実在の医療機関名・具体的受診日などの固有事実は断定して創作しない（一般表現／相対表現にする）。

### 6.6 UI再設計 ＋ 利用者データ保存（A案）
> 追加: 2026-06-16（吉本決定）。詳細仕様: [`docs/specs/ui-redesign-and-client-storage.md`](docs/specs/ui-redesign-and-client-storage.md)。

- **F10 利用者軸での保存とUI刷新**：利用者(clients)ごとに書類(documents)を保存し、ホーム（横断ハブ）・
  利用者詳細・救済→確認→カイポケ→人保存の一本動線を、editorialなデザインで提供する。
  - **A案＝要配慮個人情報をDB保存**するため、**仮名化（DBも記号＋対応表分離）・RLS（Clerk org）・保持期間・
    データ処理委託契約**と一体（ロードマップR2）。**実データ投入は委託契約／同意の締結後**（§12 Never）。
  - Claude API へは**記号のみ**送信（実名・対応表は送らない）。保持期間は**5年**（吉本決定 2026-06-16）。

## 7. 個人情報・セキュリティ・法務（介護ドメイン必須）

介護記録は **要配慮個人情報（健康・要介護状態等）** を含む。最優先で扱う。

### 7.1 方針
- **最小化**：AIに渡す個人情報は必要最小限。可能な範囲で**仮名化**（氏名→記号）してから送信し、表示時に復元。
- **非保持**：生成・評価のために一時取得したファイルは処理後に**削除**（現行の評価機能はBlobを評価後に削除済み＝この方針を踏襲）。
- **保存する場合**：Supabase の **RLS（行レベルセキュリティ）必須**、暗号化、アクセス制御、保持期間の設定。
- **送信先の明示**：Claude API（海外）へ送る範囲を明確化し、コンサル先の同意・データ取扱規程に従う（※要確認）。
- **資格情報を扱わない**：ブラウザ拡張は介護ソフトの**ログイン情報を保存・外部送信しない**。利用者の既存ログインセッション上で動く。

### 7.2 法務・規約（※要確認）
- 対象介護ソフトの**利用規約**で、ブラウザ拡張による自動操作が許容されるか確認する。NGなら API 連携や別方式へ切替。
- 個人情報保護法上の**第三者提供・委託**の整理（コンサル先＝個人情報取扱事業者、本プロダクト＝委託先の構図）。
- 必要に応じて**データ処理に関する合意書**をコンサル先と締結。

## 8. リスクと未確認事項（正直な現状）

| # | リスク／未確認 | 影響 | 対応方針 |
|---|---|---|---|
| R1 | 対象介護ソフトが未確定・複数 | 自動入力(P2)の作り込みが定まらない | P1を先行。P2は対象1つを確定してから着手。アダプタ方式で差し替え可能に |
| R2 | ソフトの利用規約が自動操作を禁止する可能性 | P2が実施不可になりうる | 着手前に規約確認（§7.2）。NG時はAPI連携/コピペ運用へ |
| R3 | 画面構造の変更で拡張が壊れる | 保守コスト | アダプタ分離、壊れたら検知できるテスト |
| R4 | 個人情報の海外送信への懸念 | 導入可否に直結 | 仮名化・最小化・合意書（§7） |
| R5 | AI生成の誤り（ハルシネーション） | 品質・信頼 | 「不明は創作せず要確認と明示」をルール化。**人間承認を必須**に。評価機能で回帰チェック |
| R6 | 各帳票の正式様式・項目の詳細 | 生成精度 | コンサル先の現物様式で確定（§3.1） |

## 9. テスト戦略

- **ユニットテスト（vitest）**：`lib/generation/` `lib/rules/` の純粋ロジック。カバレッジ目標80%（横断規約準拠）。
- **生成品質テスト**：代表的な入力サンプル → 期待される構造/必須項目の充足を検証。**評価エンジン(D)を生成物の自動採点に転用**し、品質の回帰を検知。
- **E2E（Playwright・P2以降）**：管理画面の主要フロー、拡張の入力フロー（モックサイトで）。
- **手動・実証テスト**：コンサル先での実利用フィードバック（1社実証）。

## 12. 境界（Boundaries：常にやる／先に確認／絶対やらない）

### ✅ 常にやる（Always）
- AI出力は**下書き扱い**。確定・提出の前に**必ず人間がレビュー・承認**する。
- 個人情報は最小化・可能なら仮名化してから送信。一時ファイルは処理後に削除。
- Supabase は RLS と認証を維持。
- 不明・情報不足は**創作せず「要確認」と明示**して生成する（例外＝救済モード §6.5：完成形まで埋めるが、「下書き＝人間が事実照合して確定」の原則は維持）。
- コード変更時は関連 docs と本 SPEC を同じ PR で更新（Doc-as-Code）。

### ⚠️ 先に確認（Ask first）
- 介護ソフトへの**本番データへの自動書き込み／確定**を行う運用。
- 新しい介護ソフトへの対応追加。
- 個人情報の**保存方針の変更**・外部送信先の追加。
- 破壊的操作（ファイル削除・git reset・force push）。

### ❌ 絶対やらない（Never）
- 介護ソフトの**ログイン情報を保存・外部送信**する。
- 利用者の同意なく個人情報を学習・蓄積する。
- 利用規約に反する自動化を無断で本番運用する。
- **人間の承認なしに帳票を確定・提出**する。
- lint/型チェック/フックを `--no-verify` 等で回避する。
